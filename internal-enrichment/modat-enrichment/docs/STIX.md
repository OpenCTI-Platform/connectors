# Modat STIX Output

This document describes the STIX 2.1 objects currently generated by the `modat-enrichment` connector and sent to OpenCTI.

The exact bundle content depends on:
- the input observable (must be `ipv4-addr`)
- the response from `GET /host/{ip}/v1`
- the TLP markings present on the input observable
- whether `MODAT_INCLUDE_CVES` is enabled

## High-Level Behavior

The connector starts from a single `ipv4-addr` observable already present in the incoming enrichment payload.

It then:
- updates that observable in-place through OpenCTI STIX extensions
- appends additional STIX SDOs and SCOs for ASN, organization, locations, domains, certificates, and (optionally) vulnerabilities
- appends relationships connecting these objects
- optionally appends a single summary note

A `404` from Modat (host not found) is handled silently — the connector returns the original bundle unchanged with no extra objects.

## Objects Generated

### Existing Input Observable

The incoming `ipv4-addr` observable is not recreated. It is updated with OpenCTI extension fields:

- `score` — `MODAT_DEFAULT_SCORE` (default `50`)
- `external_references` — link to `https://magnify.modat.io/hosts/<ip>`
- `labels` — always `modat`, `modat-enriched`; plus one `modat:<tag>` label for each Modat fingerprint tag returned (e.g. `modat:open-directory`, `modat:iot`)

External reference shape:

```json
{
  "source_name": "Modat (<ip>)",
  "description": "Modat Magnify host page for <ip>",
  "url": "https://magnify.modat.io/hosts/<ip>"
}
```

### Author Identity

Always appended once per bundle as the `created_by_ref` for generated objects.

- STIX type: `identity`
- Name: `Modat`
- Identity class: `organization`
- Description: `Modat Magnify enrichment source`

### Autonomous System

Created when Modat returns an integer ASN number.

- STIX type: `autonomous-system`
- Fields:
  - `number` — integer ASN
  - `name` — `AS<number>` (e.g. `AS13335`)
- OpenCTI custom properties:
  - `x_opencti_created_by_ref`
  - `x_opencti_labels` — `modat`, `modat-enriched`
  - `x_opencti_score`
  - `x_opencti_description` — set to `asn.org` when Modat provides one

Relationship created:
- `ipv4-addr --belongs-to--> autonomous-system`

### Organization Identity (ASN owner)

Created when `asn.org` is present.

- STIX type: `identity`
- Identity class: `organization`
- `name` — verbatim `asn.org` string from Modat

Relationship created:
- `autonomous-system --related-to--> identity`

The connector intentionally does **not** create a direct `ipv4-addr → identity` relationship — the chain through the AS already expresses the link.

### Country Location

Created when Modat returns `geo.country_name`.

- STIX type: `location`
- `name` and `country` set to the country name
- OpenCTI custom properties:
  - `x_opencti_location_type` = `Country`
  - `x_opencti_aliases` — `[country_iso_code]` when present

Relationship created:
- `ipv4-addr --located-at--> location` (Country)

(OpenCTI's relationship schema does not allow `autonomous-system → located-at → location`, so the AS itself is not linked to the country.)

### City Location

Created when Modat returns `geo.city_name` along with a country name.

- STIX type: `location`
- OpenCTI custom properties: `x_opencti_location_type = City`

Relationships created:
- `ipv4-addr --located-at--> location` (City)
- `location --located-at--> location` (City → Country) when both objects exist

### Domain Name (FQDN)

Created for each unique FQDN in `record.fqdns` (capped at the first 25, wildcards skipped).

- STIX type: `domain-name`
- OpenCTI custom properties:
  - `x_opencti_created_by_ref`
  - `x_opencti_score`
  - `x_opencti_labels` — `modat`, `modat-enriched`

Relationship created:
- `domain-name --resolves-to--> ipv4-addr`

### X.509 Certificate

Created for each unique TLS certificate observed across services (deduplicated by SHA-256, then serial number, then transport+port).

- STIX type: `x509-certificate`
- Fields populated when present:
  - `hashes` — `SHA-256`, `SHA-1`
  - `serial_number`
  - `issuer` — flattened RFC 4514-style DN string
  - `subject` — same shape
  - `validity_not_before` (from `tls.valid_from`)
  - `validity_not_after` (from `tls.expires_at`)
  - `is_self_signed`
  - `signature_algorithm` — parsed from `tls.raw` text dump when available
  - `subject_public_key_algorithm` — same source
- OpenCTI custom properties:
  - `x_opencti_created_by_ref`
  - `x_opencti_score`
  - `x_opencti_labels` — `modat`, `modat-enriched`

Relationship created:
- `ipv4-addr --related-to--> x509-certificate`

### Certificate SAN Domains

For each valid DNS name in the certificate's `subject_alt_name.dns` list, a `domain-name` SCO is created.

- STIX type: `domain-name`
- OpenCTI custom properties:
  - `x_opencti_labels` — `modat`, `modat-enriched`, `certificate-san`

Relationships created:
- `x509-certificate --related-to--> domain-name`
- `domain-name --resolves-to--> ipv4-addr`

### Vulnerability (optional)

Created **only** when `MODAT_INCLUDE_CVES=true`. Modat's CVE attributions are not validated, so the default is to skip them entirely.

When enabled, every unique CVE (across host-level `cves` and per-service `services[].cves`) becomes a `vulnerability` SDO.

- STIX type: `vulnerability`
- `name` — the CVE id (e.g. `CVE-2024-12345`)
- OpenCTI custom properties:
  - `x_opencti_labels` — `modat`, `modat-enriched`; plus `kev` when `is_kev=true`
  - `x_opencti_cvss_base_score` — when Modat provides `cvss`
  - no `x_opencti_score` is set — host-enrichment confidence does not apply to a CVE; CVSS is the right signal

Relationship created:
- `ipv4-addr --related-to--> vulnerability`

### Summary Note

Created when `MODAT_CREATE_NOTE=true` (default).

- STIX type: `note`
- `abstract` — `Modat Results for <ip>`
- `object_refs` — the original `ipv4-addr`
- Custom properties: `note_types = ["external"]`

The note content is a markdown summary covering:
- Host overview (IP, Magnify link, ASN/org, location, counts, anycast flag, Modat tags)
- Per-service block — transport/port (`last_scanned_port` plus alternates), protocol, fingerprint product+version, technologies, banner, HTTP title/status/headers or SSH `hassh`/`server_id`, TLS one-liner (CN, issuer CN, versions, validity dates, self-signed flag) and full SHA-256 fingerprint
- Domains list (capped at 20)
- CVE list when `MODAT_INCLUDE_CVES=true` (capped at 25)

Service rendering is capped by `MODAT_MAX_SERVICES_IN_SUMMARY` (default `25`); STIX object generation is **not** capped — every service with a TLS certificate still produces an `x509-certificate`.

## Relationship Types Used

The connector creates these STIX relationship types:

- `belongs-to` — IPv4 → AS
- `related-to` — AS → Org Identity, IPv4 → X.509, X.509 → SAN domain, IPv4 → Vulnerability
- `resolves-to` — Domain → IPv4 (for both FQDNs and SAN domains)
- `located-at` — IPv4 → Country, IPv4 → City, City → Country

## OpenCTI-Specific Fields

On generated objects:
- `x_opencti_created_by_ref`
- `x_opencti_labels`
- `x_opencti_score`
- `x_opencti_description` (on AS)
- `x_opencti_location_type`
- `x_opencti_aliases`
- `x_opencti_cvss_base_score` (on Vulnerability)

On the existing observable, via `OpenCTIStix2.put_attribute_in_extension(...)`:
- `score`
- `external_references`
- `labels`

## Example Bundle Shape

Simplified — not a byte-for-byte rendering:

```json
{
  "type": "bundle",
  "objects": [
    { "type": "identity", "name": "Modat", "identity_class": "organization" },
    { "type": "autonomous-system", "number": 13335, "name": "AS13335",
      "x_opencti_description": "Cloudflare, Inc." },
    { "type": "relationship", "relationship_type": "belongs-to",
      "source_ref": "ipv4-addr--…", "target_ref": "autonomous-system--…" },
    { "type": "identity", "name": "Cloudflare, Inc.", "identity_class": "organization" },
    { "type": "relationship", "relationship_type": "related-to",
      "source_ref": "autonomous-system--…", "target_ref": "identity--…" },
    { "type": "location", "name": "United States",
      "x_opencti_location_type": "Country" },
    { "type": "relationship", "relationship_type": "located-at",
      "source_ref": "ipv4-addr--…", "target_ref": "location--…" },
    { "type": "domain-name", "value": "example.org" },
    { "type": "relationship", "relationship_type": "resolves-to",
      "source_ref": "domain-name--…", "target_ref": "ipv4-addr--…" },
    { "type": "x509-certificate",
      "hashes": { "SHA-256": "…" },
      "issuer": "CN=Example CA",
      "subject": "CN=example.org",
      "signature_algorithm": "ecdsa-with-SHA384" },
    { "type": "relationship", "relationship_type": "related-to",
      "source_ref": "ipv4-addr--…", "target_ref": "x509-certificate--…" },
    { "type": "note", "abstract": "Modat Results for 1.1.1.1" }
  ]
}
```

## Capturing The Real Bundle

The most reliable way to inspect the exact STIX sent to OpenCTI is to monkeypatch the helper inside a test:

```python
captured = {}

def fake_create_bundle(objects):
    bundle = {"type": "bundle", "objects": objects}
    captured["bundle"] = bundle
    return bundle

helper.stix2_create_bundle = fake_create_bundle
helper.send_stix2_bundle = lambda bundle: [bundle]

connector.process_message(payload)

assert captured["bundle"]["type"] == "bundle"
assert any(obj.get("type") == "x509-certificate" for obj in captured["bundle"]["objects"])
```
